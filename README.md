# Wellness+ (React Native — iOS & Android)

Implements the screens and architecture defined in `Wellness+ Frontend &
Backend Spec`. This is real, runnable TypeScript/React Native source — not
a mockup. Because React Native is cross-platform, **the entire `src/` tree,
`App.tsx`, and root config files are shared between iOS and Android
unchanged** — only the native `ios/` and `android/` folders differ, and
those differences are native build plumbing, not application logic.

Neither native scaffold ships fully generated, because both can only be
correctly produced by their respective platform CLIs/IDEs (Xcode + Ruby/
CocoaPods for iOS; the Android Gradle toolchain for Android) running on a
machine that has them installed, which this build environment does not.
**What you get instead:** every hand-written native file that actually
matters (permissions, font linking, build config, app identity), plus
exact instructions for generating the boilerplate scaffold around them.

## What's real vs. what you generate

| This repo | Generated by you, once |
|---|---|
| All of `src/` (screens, components, services, store, types, theme) | `ios/*.xcodeproj`, `ios/Pods/`, `ios/AppDelegate.swift` |
| `App.tsx`, `index.js`, `app.json` | `android/gradlew`, `android/gradle/wrapper/gradle-wrapper.jar` (binary) |
| `package.json`, `tsconfig.json`, `babel.config.js`, `metro.config.js` | `android/app/src/main/res/mipmap-*/` (launcher icon assets) |
| `ios/Podfile`, `ios/Info.plist.additions.xml` | iOS: `ios/WellnessPlus.xcworkspace` |
| `android/build.gradle`, `android/app/build.gradle`, `android/settings.gradle`, `android/gradle.properties` | Android: the rest of the Gradle wrapper |
| `android/app/src/main/AndroidManifest.xml`, `MainActivity.kt`, `MainApplication.kt`, `res/values/*.xml` | — |

## iOS setup (on a Mac, with Xcode and CocoaPods installed)

```bash
npm install

npx react-native@0.74.2 init WellnessPlusScaffold --version 0.74.2 --skip-install
# Copy this repo's ios/Podfile and ios/Info.plist.additions.xml content
# into the generated ios/ folder (merge plist keys; Podfile can replace
# the generated one outright). Then:

cd ios && pod install && cd ..
npm run ios
```

## Android setup (on any OS, with Android Studio / SDK + JDK 17 installed)

```bash
npm install

npx react-native@0.74.2 init WellnessPlusScaffold --version 0.74.2 --skip-install
# This repo's android/ folder already contains every file that carries
# real configuration (build.gradle x2, settings.gradle, gradle.properties,
# AndroidManifest.xml, MainActivity.kt, MainApplication.kt, res/values/).
# Copy ONLY the generated project's gradlew, gradlew.bat, and
# gradle/wrapper/gradle-wrapper.jar (binary, can't be hand-written) into
# this repo's android/ folder — everything else here is already correct
# and should NOT be overwritten by the scaffold's generated versions.

cd android && chmod +x gradlew && cd ..
npm run android
```

If you'd rather not hand-merge either platform, the faster path for most
teams is `npx react-native init WellnessPlus --version 0.74.2` directly,
then copy this repo's `src/`, `App.tsx`, `index.js`, `app.json`, and root
config files over the freshly generated project — keeping the generated
`ios/` and `android/` folders as scaffolding, and applying this repo's
hand-written native files (the right-hand column above) on top.



## Backend config

The API client (`src/services/api.ts`) reads `WELLNESS_API_BASE_URL` and
defaults to a placeholder. Until a real gateway (per Frontend & Backend
Spec, Section 4.2) is deployed, every network call will fail — this is
expected. The app is built to degrade gracefully:

- **Reads** fall back to cache (`src/services/cache.ts`) and the screen
  shows a "last updated" timestamp instead of erroring.
- **Writes** queue locally (`src/services/offlineQueue.ts`) and retry
  automatically once `WELLNESS_API_BASE_URL` points at a live, reachable
  gateway.

Set the URL via `.env` (with `react-native-config`, not yet wired in this
pass) or by editing the default in `src/services/api.ts` directly for now.

## Architectural notes for whoever picks this up

- **Pricing logic lives in one place.** `src/services/pricingMirror.ts` is
  a client-side *mirror* of the backend pricing module's scoring function,
  used only to compute the Home screen's task list without an extra round
  trip. The actual credit percentage shown on the pula card always comes
  from the server (`GET /v1/engagement/{member_id}/credit`). If the backend
  scoring formula changes, this file must change with it — see the warning
  comment at the top of that file.
- **Offline-first is load-bearing, not optional.** Every write goes through
  `enqueueEngagementEvent()`, never a direct API call from a screen. This
  matches Frontend & Backend Spec, Section 3.5 / Section 6's non-functional
  requirements.
- **Icons:** the design spec references Tabler icon names (`ti-walk`, etc.);
  this build maps them to MaterialCommunityIcons equivalents in
  `src/components/TaskRow.tsx`'s `ICON_MAP`, since Tabler's web font isn't
  packaged for React Native by default. Swap in a Tabler RN package later
  for pixel parity with the original prototype if needed. On Android, the
  font is linked via `android/app/build.gradle`'s `vectoricons` block —
  without it, icons render as blank boxes on Android specifically even if
  iOS (linked separately via the Podfile + Info.plist) works fine. Don't
  assume "it works on iOS" means it works on Android for this dependency.
- **Permissions are requested per-platform but justified once.** Both
  `ios/Info.plist.additions.xml` and `android/app/src/main/AndroidManifest.xml`
  request location (clinic locator) and, on Android, explicit
  `POST_NOTIFICATIONS` (required from API 33 onward — iOS doesn't have an
  equivalent manifest-level requirement, it's handled at runtime via the
  notification permission API instead). Same feature, different platform
  mechanics — both are wired to the same underlying app behavior.
- **Hermes is enabled on both platforms** (`ios/Podfile`'s
  `hermes_enabled => true`, `android/gradle.properties`'
  `hermesEnabled=true`), so `pricingMirror.ts`'s scoring math runs on
  identical JS engine semantics cross-platform — relevant given how
  precisely that function needs to match the backend's formula.
- **New Architecture (Fabric/TurboModules) is off on both platforms** for
  this pass. The dependency set (react-native-svg, vector-icons, navigation,
  netinfo, async-storage) is stable on the old architecture; revisit only
  if a future native module requires it.
- **Out of scope at this pass**, per spec Section 3.6: native HealthKit
  integration, push notification deep-linking, in-app clinic booking.

## Project structure

```
App.tsx                     Root: auth gate, navigation, queue auto-flush
src/
  screens/                  Home, Screening, Activity, Rewards, Profile, Login
  components/                PulaCard, TierProgressBar, TaskRow
  services/
    api.ts                  Real HTTP client, matches backend spec contracts
    offlineQueue.ts          Write queue, idempotent retry
    cache.ts                Read-side cache for offline screens
    pricingMirror.ts        Client-side scoring mirror (Home task list only)
  store/
    authStore.ts             Session/auth state (zustand)
    engagementStore.ts       Score/tier/credit state (zustand)
  theme/tokens.ts            Design tokens — matches spec Section 2 exactly
  types/api.ts               TypeScript types matching all API contracts
  navigation/                Bottom tab navigator, 5 destinations
ios/
  Podfile                   Native dependency list (merge into generated ios/)
  Info.plist.additions.xml  Plist keys to merge into generated Info.plist
android/
  build.gradle               Project-level Gradle config (SDK/tooling versions)
  settings.gradle            Module registration, autolinking entry point
  gradle.properties          Hermes, AndroidX, architecture flags
  app/
    build.gradle              App module config + vector-icons font linking
    src/main/
      AndroidManifest.xml      Permissions (location, network, notifications)
      java/com/wellnessplus/
        MainActivity.kt         RN activity entry point
        MainApplication.kt      RN host bootstrap + native module autolinking
      res/values/
        strings.xml              App display name
        styles.xml               Launch theme
        colors.xml               Launch background (matches theme/tokens.ts)
```

## Platform parity

Both platforms run the exact same `src/` logic, including the offline
queue, cache layer, and pricing mirror — there is no Android-specific or
iOS-specific application code anywhere in this repo. If a bug only
reproduces on one platform, the cause is almost certainly in the native
layer (permissions, font linking, Gradle/Pod config) covered above, not in
`src/`.
